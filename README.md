# 알고먹자 - Java Swing 식단 관리 프로그램

## 프로젝트 소개

알고먹자는 Java Swing 기반의 데스크톱 식단 및 건강 관리 프로그램입니다.  
사용자는 회원가입과 로그인을 통해 식단, 음식 정보, 운동 정보, 공지사항 등을 확인할 수 있으며, 관리자는 공지사항을 관리할 수 있습니다.

이 프로젝트는 기존 Eclipse 기반 Java 프로젝트를 macOS 환경에서 복구하고, MySQL 연동 및 실행 오류를 수정하여 포트폴리오용으로 정리한 프로젝트입니다.

## 주요 기능

### 사용자 기능

#### 회원가입 및 로그인

- 사용자는 아이디, 비밀번호, 이름, 전화번호, 이메일, 생년월일, 성별 정보를 입력하여 회원가입할 수 있습니다.
- 회원가입 시 아이디, 이메일, 전화번호 중복 여부를 확인합니다.
- 로그인 성공 시 사용자 세션을 저장하여 이후 화면에서 로그인 사용자의 정보를 활용합니다.
- 신규 비밀번호는 사용자별 16바이트 salt와 600,000회 반복을 적용한 `PBKDF2WithHmacSHA256` 형식으로 저장합니다.
- 기존 평문 일반 사용자·관리자 계정은 로그인 성공 시 PBKDF2 형식으로 점진 전환하며, 손상된 PBKDF2 형식은 평문으로 재해석하지 않습니다.
- 로그인 성공 객체와 세션에는 DB 비밀번호나 저장 해시를 보관하지 않습니다.
- 로그인 실패 시 아이디는 유지하고 비밀번호만 지우며, 로그인 성공과 로그아웃 시 비밀번호와 보기 상태를 초기화합니다.
- 아이디 저장을 선택하면 로그아웃 후 저장 아이디를 다시 표시하고, 선택하지 않으면 아이디도 비웁니다.
- 미구현 이메일 계정 찾기 UI는 제거했으며 이메일 가입·중복검사·회원정보 기능은 유지합니다.
- 로그인과 회원가입·재설정·회원정보 비밀번호 필드는 보기·숨김을 지원합니다.
- 새 비밀번호 필드는 기존 검증 정책이 허용하는 ASCII 문자와 최대 20자 입력을 직접 입력·붙여넣기에 동일하게 적용하고, 제출 시 최종 규칙을 다시 검증합니다.
- 이름과 휴대폰 번호로 아이디를 찾을 수 있습니다.
- 이름·아이디·휴대폰 번호로 사용자를 확인한 뒤 기존 비밀번호를 노출하지 않고 새 비밀번호를 설정할 수 있습니다.
- 회원정보에서 일반 정보만 바꿀 때는 현재 비밀번호를 요구하지 않으며, 새 비밀번호를 바꿀 때는 현재 비밀번호 검증과 확인값·규칙 검증을 모두 통과해야 합니다.
- 비밀번호 변경 요청의 검증이 하나라도 실패하면 일반 정보까지 저장하지 않는 fail-closed 정책을 적용하며, 성공 경로는 하나의 조건부 `UPDATE` 뒤에만 세션 객체를 갱신합니다.

#### 신체정보 입력

- 회원가입 후 키, 몸무게, 체지방률, 체지방량, 골격근량 등의 신체정보를 입력할 수 있습니다.
- 신체정보 입력을 원하지 않는 경우 Skip 버튼을 통해 메인 화면으로 이동할 수 있습니다.
- 기존에는 Skip 버튼 클릭 시 로그인 화면으로 이동하는 문제가 있었으며, 이를 사용자 메인 화면으로 이동하도록 수정했습니다.

#### 음식 정보 조회 및 검색

- 음식 목록을 조회할 수 있습니다.
- 음식 이름을 기준으로 검색할 수 있습니다.
- 음식별 칼로리, 탄수화물, 단백질, 지방 정보를 확인할 수 있습니다.

#### 식단 기록 및 영양소 조회

- 아침, 점심, 저녁, 간식 유형을 선택해 음식을 검색하고 상세 정보와 중량을 확인한 뒤 식단을 저장할 수 있습니다.
- 식단 카드와 음식 검색·상세·담은 음식 목록은 440px 고정 폭에서 정보와 조작 영역을 구분하고 세로 스크롤과 긴 음식명 표시를 지원합니다.
- 식사 유형이 없거나 저장 과정이 실패한 경우 불완전한 저장과 잘못된 성공 안내를 차단합니다.
- 식단 헤더와 여러 상세 음식은 같은 JDBC `Connection`의 단일 트랜잭션으로 저장하며 전부 성공할 때만 `commit`합니다.
- 저장 성공 후 식단 기록 화면으로 복귀하며 해당 식사 카드의 열량이 갱신됩니다.
- 식사별 음식 정보를 기반으로 칼로리, 탄수화물, 단백질, 지방 섭취량을 계산할 수 있습니다.
- 오늘 섭취한 영양소 정보를 조회할 수 있습니다.

#### 운동 정보 조회

- 운동 정보를 카테고리별로 조회할 수 있습니다.
- 전체 운동 목록을 조회할 수 있습니다.
- 운동 목록에서 운동명 또는 운동 유형을 기준으로 검색할 수 있으며, 검색어를 지우면 기존 목록으로 복구됩니다.
- 운동을 선택한 뒤 운동 시간과 현재 체중을 입력하여 예상 소모 칼로리를 계산할 수 있습니다.
- 계산된 운동 기록을 저장할 수 있습니다.
- 월간 캘린더에서 이전·다음 달로 이동하고 오늘과 선택 날짜를 구분할 수 있습니다.
- 선택 날짜의 운동 기록과 식단 기록을 함께 확인할 수 있으며 기록이 없으면 별도 안내가 표시됩니다.

#### 체중 목표 관리

- 시작 체중, 현재 체중, 목표 체중, 목표 기간과 기간 경과를 구분해 확인할 수 있습니다.
- 목표를 설정·수정한 직후와 목표 화면에 다시 진입할 때 최신 목표 데이터를 다시 조회합니다.
- 감량·증량·반대 방향 변화·목표 달성 상태를 구분하고, 화면 진행률은 `NaN`, 무한대, 음수와 100% 초과 폭이 생기지 않도록 0~100 범위로 제한합니다.
- 목표 데이터가 없을 때는 이전 수치가 남지 않는 빈 상태와 명시적인 목표 설정 버튼을 표시합니다.

#### 공지사항 조회

- 사용자는 카드 형태로 구분된 공지사항 목록을 확인할 수 있습니다.
- 공지사항 제목, 작성자, 작성일 정보를 확인할 수 있습니다.
- 다건 목록 세로 스크롤, 카드 전체 클릭, 긴 제목 툴팁을 지원합니다.
- 공지사항 상세에서 작성자와 전체 작성 날짜, 본문을 확인하고 사용자 공지 목록으로 돌아갈 수 있습니다.
- 긴 본문은 줄바꿈과 세로 스크롤로 확인할 수 있습니다.
- 여러 첨부파일의 긴 이름을 목록과 툴팁으로 확인하고 앱 저장 복사본을 열 수 있습니다.

### 관리자 기능

#### 관리자 로그인

- 일반 사용자와 관리자를 구분하여 로그인합니다.
- 관리자 계정은 별도의 `admin` 테이블을 통해 관리됩니다.
- 공개 저장소에는 고정 관리자 계정이나 비밀번호 seed를 포함하지 않습니다. 관리자 계정은 로컬 환경에서 PBKDF2 인코딩 값으로 별도 준비해야 합니다.

#### 공지사항 관리

- 관리자는 다건 공지 목록과 상세 내용을 확인하고 공지를 작성·수정·단일 또는 다중 선택 삭제할 수 있습니다.
- 첨부파일 없이도 공지를 저장할 수 있으며 작성·수정·삭제 후 목록이 갱신됩니다.
- 공지사항 첨부파일 관리를 위한 `notice_files` 테이블 구조를 포함하고 있습니다.
- 목록의 긴 제목은 말줄임과 전체 제목 툴팁으로 표시하고 상세의 긴 본문은 줄바꿈과 세로 스크롤을 지원합니다.
- 목록·상세·작성·수정·삭제와 첨부 복사는 `SwingWorker`로 처리해 DB·파일 작업 중 Swing 이벤트 처리가 장시간 멈추지 않게 했습니다.
- 첨부는 원본 표시 이름과 UUID 저장 이름을 분리하고 크로스플랫폼 `Path`·`Files` API로 앱 전용 저장소에 복사합니다.
- 공지 삭제 후에는 첨부 DB 레코드와 앱 전용 UUID 저장 복사본을 정리하며, 저장소 밖 경로나 사용자가 선택한 원본 파일은 삭제하지 않습니다.

### 데이터베이스 연동 기능

- MySQL 데이터베이스와 JDBC를 통해 연동합니다.
- DB 접속 정보는 `src/db.properties` 파일로 분리했습니다.
- 실제 비밀번호가 포함된 `db.properties`는 Git 추적에서 제외하고, 예시 파일인 `db.properties.example`만 제공합니다.
- `sql/schema.sql`을 통해 데이터베이스, 테이블, 테스트 데이터를 재생성할 수 있도록 정리했습니다.
- `DBConnectionMgr`의 제거 예정 `finalize()`와 `Class.newInstance()` 의존을 제거하면서 기존 커넥션 풀·DAO 호출 계약은 유지했습니다.

### 복구 및 개선한 기능

- Windows 환경 기준으로 설정되어 있던 MySQL Connector 경로 문제를 macOS 환경에 맞게 수정했습니다.
- 프로젝트 폴더의 read-only 권한 문제를 해결하여 Eclipse 빌드가 가능하도록 복구했습니다.
- 누락된 MySQL 테이블을 DAO 코드 기준으로 분석하여 생성했습니다.
- 관리자 로그인, 일반 사용자 회원가입, 음식/운동/공지사항 조회 흐름을 확인했습니다.
- 신체정보 입력 화면에서 Skip 버튼 클릭 시 로그인 화면으로 돌아가던 화면 전환 문제를 수정했습니다.
- 운동 목록 화면에서 카테고리 선택 화면으로 돌아갈 수 있도록 흐름을 개선했습니다.
- `유산소&코어` 카테고리 조회 시 유산소와 코어 운동을 함께 조회하도록 보정했습니다.
- 전체 운동 목록 조회 기능을 추가했습니다.
- 운동 목록 검색 후 검색어를 삭제하면 원래 목록이 복구되도록 개선했습니다.
- 운동 칼로리 입력 화면에서 운동 변경 시 이전 입력값이 남지 않도록 초기화했습니다.
- 운동 기록 저장 SQL과 운동명 저장 흐름을 실제 `exercise_log` 구조에 맞게 보정했습니다.
- 사용자 메인 화면의 네비게이션 초기화 순서를 보정했습니다.
- 음식 검색부터 상세·중량 조절·담기·식단 저장·홈 복귀까지의 흐름을 안정화했습니다.
- 식단 홈 카드와 음식 검색·선택·담은 음식 목록의 폭, 정보 계층, 조작 버튼과 스크롤을 개선했습니다.
- 식단 헤더와 상세 저장을 하나의 JDBC 트랜잭션으로 통합해 부분 저장 위험을 줄였습니다.
- 월간 캘린더와 선택 날짜의 식단·운동 기록 조회를 구현했습니다.
- 로그인 비밀번호 보기·숨김, 휴대폰 기반 아이디 찾기와 본인 확인 후 비밀번호 재설정을 구현했습니다.
- 사용자 공지 다건 목록·긴 본문과 관리자 공지 작성·수정·단일·다중 삭제 흐름을 실제 실행으로 검증했습니다.
- 로그인 전 세션이 없는 정상 상태의 반복 콘솔 로그를 제거하고 실제 오류·예외 진단 로그는 유지했습니다.
- 일반 사용자와 관리자 비밀번호 저장·검증을 PBKDF2로 전환하고 기존 평문 계정의 점진 전환을 적용했습니다.
- 회원가입·비밀번호 재설정·회원정보 비밀번호 변경의 최종 규칙 검증과 해시 저장을 통일했습니다.
- 사용자 객체와 세션에서 비밀번호·저장 해시 필드를 제거했습니다.
- 정상 흐름에서 출력되던 사용자 식별값, 신체·운동 입력값, 공지 내용과 로컬 파일 경로 로그를 제거했습니다.
- 인증·계정 찾기·회원정보·목표·사용자 및 관리자 공지 화면에 작은 공통 테마를 적용하고 주요 안내창의 부모를 현재 앱 창으로 통일했습니다.

## 화면 크기 및 개인정보 필드 정책

- 현재 데스크톱 UI는 `440×956` 고정 크기이며 `setResizable(false)`를 유지합니다.
- 레거시 `null` layout 화면이 다수 남아 있어 자유 크기 조절과 반응형 배치는 지원하지 않습니다.
- 이번 UI 고도화에서는 주요 인증·회원정보·목표·공지 화면의 잘림, 겹침, 스크롤과 스타일 일관성을 고정 크기 안에서 개선했습니다.
- 향후 전체 화면을 레이아웃 매니저 기반으로 전환하면 반응형 창 크기 지원을 검토할 수 있습니다.
- 휴대폰 번호는 현재 아이디 찾기와 비밀번호 재설정의 가입 정보 대조에 사용합니다. 실제 SMS 소유 인증은 구현하지 않았습니다.
- 생년월일은 현재 회원가입 시 저장되지만 실제 나이 또는 권장 칼로리 계산에는 사용하지 않습니다. 향후 계산 기능에 연결하거나 DB 마이그레이션과 함께 제거할 후속 항목입니다.

## 프로젝트 실행 방법

이 프로젝트는 Eclipse 기반 Java Swing 프로젝트입니다.  
실행을 위해서는 JDK, MySQL Server, MySQL Connector/J 설정이 필요합니다.

### 1. JDK 설치 확인

JDK 21 이상을 설치한 뒤 터미널에서 아래 명령어로 확인합니다.

```bash
java -version
javac -version
```

### 2. MySQL Server 설치 및 실행 확인

MySQL Community Server 8.0 이상을 설치합니다.

macOS에서 MySQL 공식 설치 파일을 사용한 경우, 터미널에서 아래 명령어로 접속할 수 있습니다.

```bash
/usr/local/mysql/bin/mysql -u root -p
```

비밀번호 입력 후 아래처럼 표시되면 정상 접속입니다.

```text
mysql>
```

### 3. 데이터베이스 및 테이블 생성

MySQL에 접속한 뒤, 프로젝트에 포함된 `sql/schema.sql` 파일을 실행합니다.

예시:

```sql
SOURCE /Users/사용자명/Downloads/project8/sql/schema.sql;
```

또는 `schema.sql` 파일 내용을 MySQL 콘솔에 직접 붙여넣어 실행할 수 있습니다.

`schema.sql` 실행 시 다음 항목이 생성됩니다.

- `almja` 데이터베이스
- `admin` 테이블
- `user` 테이블
- `food` 테이블
- `exercise` 테이블
- `exercise_log` 테이블
- `notice` 테이블
- `notice_files` 테이블
- `meal` 테이블
- `meal_log` 테이블
- 기본 음식·운동 테스트 데이터

### 4. DB 설정 파일 생성

`src/db.properties.example` 파일을 복사하여 `src/db.properties` 파일을 생성합니다.

예시:

```properties
db.driver=com.mysql.cj.jdbc.Driver
db.url=jdbc:mysql://localhost:3306/almja?characterEncoding=UTF-8&serverTimezone=UTC
db.user=root
db.password=your_password_here
```

`db.password`에는 본인의 MySQL root 비밀번호를 입력합니다.

보안상 `src/db.properties`는 `.gitignore`에 포함되어 있으며 GitHub에 업로드하지 않습니다.

### 5. Eclipse에서 프로젝트 Import

Eclipse에서 아래 순서로 프로젝트를 가져옵니다.

1. `File`
2. `Import`
3. `General`
4. `Existing Projects into Workspace`
5. 프로젝트 루트 폴더 선택
6. `Finish`

### 6. MySQL Connector/J 설정

프로젝트에 MySQL Connector/J 라이브러리가 필요합니다.

Eclipse에서 아래 순서로 `.jar` 파일을 추가합니다.

1. 프로젝트 우클릭
2. `Properties`
3. `Java Build Path`
4. `Libraries`
5. `Classpath` 선택
6. `Add External JARs...`
7. `mysql-connector-j-x.x.x.jar` 선택
8. `Apply and Close`

현재 복구 환경에서는 `mysql-connector-j-9.7.0.jar`를 사용했습니다.

### 7. 프로젝트 실행

Eclipse에서 아래 파일을 실행합니다.

```text
src/main/MainFrame.java
```

실행 방법:

1. `MainFrame.java` 우클릭
2. `Run As`
3. `Java Application`

실행에 성공하면 `알고먹자` 로그인 화면이 표시됩니다.

## 현재 한계 및 후속 과제

이 프로젝트는 주요 사용자·관리자 흐름의 Eclipse 실행 검증을 마쳤지만 다음 항목은 별도 확장 또는 운영 결정이 필요합니다.

- 아이디·비밀번호 찾기의 휴대폰 확인은 가입 정보 대조 방식이며 실제 SMS 인증 서비스는 연동하지 않았습니다.
- 생년월일을 실제 나이·권장 칼로리 계산에 연결할지, DB 마이그레이션과 함께 제거할지 결정해야 합니다.
- 기존 평문 계정의 점진 전환을 위해 평문 호환 인증을 임시 유지합니다. 전환 완료를 확인한 뒤 제거 시점을 정해야 합니다.
- 관리자 계정 생성·비밀번호 재설정 UI가 없습니다. 공개 seed 대신 배포 환경별 안전한 프로비저닝 절차가 필요합니다.
- 식단 저장의 정상 다건 `commit`은 실행 검증했지만, DB나 스키마를 훼손하지 않고 상세 저장만 실패시키는 경로가 없어 실제 장애 상황의 `rollback`은 정적 검증만 완료했습니다.
- `DBConnectionMgr`의 raw `Vector.addElement`에서 unchecked 경고 2건이 남아 있습니다. 전체 커넥션 풀 제네릭화는 별도 리팩터링 범위입니다.
- 레거시 `null` layout을 전체 레이아웃 매니저 기반으로 전환한 뒤 반응형 창 크기 지원을 검토해야 합니다.
- `sql/schema.sql`의 관리자 seed 제거는 기존 로컬 DB 데이터를 변경하지 않습니다. 이미 노출된 자격 증명을 사용했다면 별도 폐기·교체가 필요합니다.
