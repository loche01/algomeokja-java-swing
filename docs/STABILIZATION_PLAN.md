# 알고먹자 통합 안정화 계획

이 계획은 `codex-stabilization` 브랜치에서 순서대로 수행한다. 각 작업은 관련 파일과 호출 흐름을 먼저 분석하고 최소 범위로 수정하며, 기능별 한국어 커밋과 `docs/CODEX_WORKLOG.md` 기록을 남긴다. DB 스키마 변경, 실제 DB 확인, Eclipse 화면 확인 또는 각 체크포인트가 필요한 시점에는 자동 진행을 중단한다.

## 0단계: 네비게이션 초기화

### 작업 범위

- 네비게이션 바 생성 후 `HomeDaily` 표시
- 초기화 경고 제거
- `MainUserPanel` 최소 수정

### 목표

- `MainUserPanel` 생성 과정에서 네비게이션 바가 먼저 준비되고 초기 홈 화면이 안정적으로 표시되도록 초기화 순서를 바로잡는다.
- 기존 패널 이름, 화면 전환 구조와 `MainFrame`의 440x956 고정 크기를 유지한다.

### 예상 수정 파일

- `src/main/MainUserPanel.java`

### 완료 기준

- 네비게이션 바가 생성된 뒤 `HomeDailyPanel`이 초기 화면으로 표시되는 호출 순서가 정적 분석으로 확인된다.
- 초기화되지 않은 필드 접근 경고의 원인이 제거된다.
- 다른 패널의 등록 및 전환 방식에는 변경이 없다.
- `git diff --check`가 통과하고 관련 Java 정적 컴파일 검증에서 새 오류가 없다.

### 예상 커밋

- `수정: 사용자 메인 네비게이션 초기화 순서 보정`

### 중단 조건

- 수정이 `MainUserPanel` 밖의 화면 전환 구조까지 확장되는 경우
- 현재 작업과 무관한 컴파일 오류가 발견되는 경우
- Eclipse에서만 재현하거나 확인할 수 있는 초기 화면 문제가 남는 경우

### 사용자 테스트 항목

- 일반 사용자 로그인 후 네비게이션 바와 홈 일일 화면이 함께 표시되는지 확인
- 최초 화면에서 빈 패널, 중복 패널 또는 화면 깜박임이 없는지 확인
- 네비게이션 버튼으로 기존 화면 전환이 유지되는지 확인

## 1단계: 식단 저장 흐름 안정화

### 작업 범위

- 음식 검색 -> 상세 -> 담기 -> 저장 흐름 점검
- 식사 유형 누락 저장 차단
- 존재하지 않는 `showPanel("Home")` 호출 보정
- 저장 성공 후 올바른 화면 이동
- 오류 안내와 예외 처리 보강

### 목표

- 음식 검색부터 식단 저장과 홈 복귀까지 사용자 흐름이 끊기지 않도록 화면 전환과 입력 검증을 안정화한다.
- 식사 유형이 없는 불완전한 기록의 저장을 UI에서 차단하고, 저장 실패를 성공으로 안내하지 않는다.

### 예상 수정 파일

- `src/main/MainUserPanel.java`
- `src/panel/FoodListPanel.java`
- `src/panel/FoodInfoPanel.java`
- `src/panel/MealRecordAdd.java`
- `src/panel/MealRecordPanel.java`
- 관련 호출 흐름 분석 결과 필요한 기존 DAO 또는 모델 파일

### 완료 기준

- 음식 검색 결과에서 상세 화면으로 이동하고 선택한 음식을 담을 수 있다.
- 식사 유형을 선택하지 않으면 저장이 실행되지 않고 명확한 안내가 표시된다.
- 등록되지 않은 패널 이름을 사용하는 화면 전환 호출이 없다.
- 저장 성공 후 등록된 홈 화면으로 이동하고, 실패 시 현재 입력 흐름을 유지하며 오류를 안내한다.
- `git diff --check`와 관련 Java 정적 컴파일 검증이 통과한다.

### 예상 커밋

- `수정: 식사 유형 누락 저장 차단`
- `수정: 식단 저장 후 화면 전환 보정`
- `수정: 식단 저장 오류 안내 보강`

### 중단 조건

- DAO와 실제 DB 스키마 불일치 때문에 코드만으로 저장 흐름을 확정할 수 없는 경우
- DB 테이블이나 컬럼 변경 또는 실제 데이터 확인이 필요한 경우
- 예상 파일을 넘어 식단 기능 전반의 구조 변경이 필요한 경우
- 체크포인트 1에 도달한 경우

### 사용자 테스트 항목

- 음식명 검색, 검색 결과 선택 및 상세 정보 표시 확인
- 음식 담기 후 선택 목록과 수량 또는 영양 정보 유지 확인
- 식사 유형 미선택 상태에서 저장 차단과 안내 문구 확인
- 식사 유형 선택 후 저장 성공 안내와 홈 복귀 확인
- 저장 실패 시 새 예외, 빈 화면 또는 잘못된 성공 안내가 없는지 확인

### 체크포인트 1

- 네비게이션
- 음식 검색
- 음식 상세
- 음식 담기
- 식단 저장
- 홈 복귀

체크포인트에 도달하면 자동 진행을 멈추고 완료 커밋, 변경 파일, 자동 검증 결과, Eclipse 확인 순서, 예상 정상 화면, 남아 있어도 되는 로그, 새 오류 판단 기준과 작업일지 상태를 보고한다.

## 2단계: 월간 캘린더 구현

### 작업 범위

- 월간 날짜 그리드
- 이전 달·다음 달 이동
- 오늘 표시
- 날짜 선택
- 선택 날짜 운동 기록
- 선택 날짜 식단 기록
- 기록 없음 안내
- 고정 창 내부 스크롤과 레이아웃

### 목표

- 440x956 고정 창 안에서 월간 달력과 선택 날짜의 식단·운동 기록을 한 화면 흐름으로 조회할 수 있게 한다.
- 월 이동, 오늘 표시와 날짜 선택 상태가 서로 일관되게 동작하도록 한다.

### 예상 수정 파일

- `src/panel/CalendarPanel.java`
- `src/DB/CalendarDAO.java`
- 선택 날짜 식단·운동 조회에 이미 사용되는 기존 DAO 및 모델 파일

### 완료 기준

- 표시 월에 맞는 날짜 그리드가 생성되고 이전 달·다음 달 이동이 가능하다.
- 오늘과 선택 날짜가 구분되어 표시된다.
- 선택 날짜의 운동 기록과 식단 기록이 각각 표시되며, 없을 때 안내가 표시된다.
- 고정 창 안에서 내용이 잘리지 않고 필요한 영역을 스크롤할 수 있다.
- DB 스키마 변경 없이 기존 조회 계약을 사용하고 `git diff --check`와 관련 정적 검증이 통과한다.

### 예상 커밋

- `기능: 월간 캘린더 날짜 탐색 구현`
- `기능: 선택 날짜 식단 및 운동 기록 표시`
- `수정: 캘린더 고정 창 레이아웃 보정`

### 중단 조건

- 선택 날짜 기록 조회에 DB 테이블이나 컬럼 변경이 필요한 경우
- 날짜 저장 형식이 코드와 스키마 사이에서 일치하지 않아 실제 DB 확인이 필요한 경우
- 고정 창 레이아웃의 정상 여부를 Eclipse 화면에서 확인해야 하는 경우
- 체크포인트 2에 도달한 경우

### 사용자 테스트 항목

- 현재 월 날짜 배열과 오늘 표시 확인
- 이전 달·다음 달 이동 및 연도 경계 이동 확인
- 날짜 선택 표시와 선택 변경 확인
- 기록이 있는 날짜의 운동·식단 내용 확인
- 기록이 없는 날짜의 안내 확인
- 긴 기록에서 스크롤 및 화면 깨짐 여부 확인

### 체크포인트 2

- 월 이동
- 날짜 선택
- 오늘 날짜
- 운동 기록
- 식단 기록
- 화면 깨짐 여부

체크포인트에 도달하면 자동 진행을 멈추고 완료 커밋, 변경 파일, 자동 검증 결과, Eclipse 확인 순서, 예상 정상 화면, 남아 있어도 되는 로그, 새 오류 판단 기준과 작업일지 상태를 보고한다.

## 3단계: 로그인 비밀번호 UX

### 작업 범위

- 비밀번호 보기/숨기기
- 비밀번호 문자 규칙 안내
- 기존 로그인 흐름 유지

### 목표

- 로그인 화면에서 비밀번호 표시 상태를 사용자가 전환할 수 있게 하고 입력 규칙을 명확히 안내한다.
- 인증 요청, 세션 생성과 로그인 성공·실패 처리에는 불필요한 변경을 하지 않는다.

### 예상 수정 파일

- `src/panel/LoginPanel.java`

### 완료 기준

- 비밀번호 보기/숨기기 전환 시 입력값과 포커스가 유지된다.
- 비밀번호 문자 규칙 안내가 고정 창 안에서 잘리지 않는다.
- 기존 로그인 호출과 성공·실패 화면 전환이 유지된다.
- 비밀번호 또는 개인정보가 콘솔에 출력되지 않는다.
- `git diff --check`와 관련 정적 컴파일 검증이 통과한다.

### 예상 커밋

- `기능: 로그인 비밀번호 표시 전환 추가`
- `개선: 로그인 비밀번호 입력 규칙 안내`

### 중단 조건

- 로그인 인증 정책 변경이 필요한 경우
- 기존 로그인 실패가 실제 DB 데이터 확인 없이는 판단되지 않는 경우
- Eclipse에서 비밀번호 필드 표시나 레이아웃 확인이 필요한 경우

### 사용자 테스트 항목

- 비밀번호 입력 후 보기/숨기기 반복 전환 확인
- 전환 후 입력값, 커서와 로그인 버튼 동작 유지 확인
- 문자 규칙 안내의 가독성과 화면 잘림 여부 확인
- 정상·오류 로그인에서 기존 안내와 화면 전환 확인

## 4단계: 아이디 및 비밀번호 찾기

### 작업 범위

- 휴대폰 기반 사용자 확인
- 아이디 찾기
- 본인 확인 후 새 비밀번호 설정
- 기존 비밀번호 노출 금지
- 입력 검증과 로그인 화면 복귀

### 목표

- 휴대폰 번호로 사용자를 확인해 아이디를 찾고, 본인 확인 뒤 기존 비밀번호를 노출하지 않은 채 새 비밀번호를 설정한다.
- 생년월일과 성별 데이터는 삭제하지 않으며 키보드 입력 언어를 강제로 변경하지 않는다.

### 예상 수정 파일

- `src/panel/FindIdPwPhone.java`
- `src/panel/LoginPanel.java`
- `src/DB/UserDAO.java`
- `src/DB/LoginDAO.java`
- 사용자 확인과 비밀번호 갱신에 이미 사용되는 기존 모델 파일

### 완료 기준

- 휴대폰 번호 입력 검증과 일치 사용자 확인이 가능하다.
- 아이디 찾기 결과는 필요한 범위로만 표시되고 개인정보가 콘솔에 출력되지 않는다.
- 비밀번호 찾기는 기존 비밀번호 조회·표시 없이 새 비밀번호와 확인값을 검증한 후 갱신한다.
- 완료 또는 취소 후 로그인 화면으로 정상 복귀한다.
- DB 스키마 변경 없이 `git diff --check`와 관련 정적 검증이 통과한다.

### 예상 커밋

- `기능: 휴대폰 기반 아이디 찾기 구현`
- `기능: 본인 확인 후 비밀번호 재설정 구현`
- `수정: 계정 찾기 입력 검증 및 화면 복귀 보강`

### 중단 조건

- 휴대폰 번호 중복 계정 처리 등 인증 정책을 코드만으로 확정할 수 없는 경우
- 비밀번호 저장 방식 또는 갱신에 DB 스키마 변경이 필요한 경우
- 실제 사용자 데이터 또는 Eclipse 화면 확인이 필요한 경우
- 개인정보나 기존 비밀번호를 노출해야만 기존 흐름을 유지할 수 있는 경우

### 사용자 테스트 항목

- 가입된 휴대폰 번호와 미가입 번호의 아이디 찾기 안내 확인
- 잘못된 형식, 빈 입력과 불일치 사용자 처리 확인
- 본인 확인 후 새 비밀번호·확인값 일치 검증 확인
- 기존 비밀번호가 어느 화면과 콘솔에도 표시되지 않는지 확인
- 새 비밀번호 설정 완료 후 로그인 화면 복귀와 로그인 확인

- `개선: 관리자 공지사항 목록 가독성 보강`

### 중단 조건

- UI 변경이 공지 DAO 또는 DB 스키마 변경으로 확장되는 경우
- 기존 작성·수정·삭제 기능의 정책 결정이 필요한 경우
- Eclipse에서 고정 창 레이아웃과 스크롤 확인이 필요한 경우
- 체크포인트 3에 도달한 경우

### 사용자 테스트 항목

- 사용자 공지 목록의 스크롤, 카드 간격과 정보 가독성 확인
- 사용자 공지 상세의 긴 본문 스크롤 확인
- 관리자 공지 목록의 스크롤과 기존 상세 진입 확인
- 관리자 공지 작성·수정·삭제 흐름이 유지되는지 확인
- 새 예외, 화면 겹침 또는 잘림이 없는지 확인

### 체크포인트 3

- 로그인
- 비밀번호 보기/숨기기
- 아이디 찾기
- 비밀번호 재설정
- 사용자·관리자 공지사항
- 새 예외와 화면 깨짐 여부

체크포인트에 도달하면 자동 진행을 멈추고 완료 커밋, 변경 파일, 자동 검증 결과, Eclipse 확인 순서, 예상 정상 화면, 남아 있어도 되는 로그, 새 오류 판단 기준과 작업일지 상태를 보고한다.
