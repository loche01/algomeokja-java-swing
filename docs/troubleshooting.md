# 주요 트러블슈팅

프로젝트 개발과 종료 후 개선 과정에서 원인 분석과 구조 변경이 필요했던 항목을 정리합니다.

## 1. 운영체제마다 달라지는 JDBC 드라이버 경로

### 문제

Eclipse의 `.classpath`에 특정 PC의 MySQL Connector/J 절대 경로가 저장되어 다른 환경에서 Build Path 오류가 발생했습니다.

### 원인

외부 JAR 경로가 Windows 또는 macOS의 사용자 디렉터리에 직접 의존했습니다.

### 해결

- 저장소 내부의 `lib/mysql-connector-j.jar` 상대 경로를 사용하도록 프로젝트 설정을 정리했습니다.
- JAR 바이너리는 Git에서 제외하고 사용자가 공식 배포본을 직접 배치하도록 했습니다.
- JDK와 드라이버 설정 절차는 DB 설정 문서로 분리했습니다.

### 결과

개발자별 절대 경로를 저장소에 기록하지 않고 동일한 프로젝트 구조로 Eclipse Build Path를 구성할 수 있습니다.

## 2. DAO와 초기 스키마의 불일치

### 문제

초기 스키마에는 신체정보와 목표 관리에 필요한 테이블이 빠져 있었고, 운동 기록과 공지 첨부파일도 DAO가 기대하는 컬럼과 차이가 있었습니다.

### 원인

팀 개발 중 변경된 SQL이 여러 보조 파일과 로컬 DB에 나뉘어 최종 설치 기준이 하나로 정리되지 않았습니다.

### 해결

- 전체 DAO의 `SELECT`, `INSERT`, `UPDATE`, `DELETE` SQL을 기준으로 테이블과 컬럼을 대조했습니다.
- `bodyinfo`, `target`, `exercise_log`, `notice_files`를 포함한 신규 환경용 구조를 `sql/schema.sql`에 통합했습니다.
- 사용자 생년월일·성별 필드와 목표 테이블의 최종 구조가 통합된 뒤 중복 보조 SQL을 제거했습니다.
- 공개 스키마에서 관리자 자격 증명 seed를 제외했습니다.

### 결과

현재 DAO가 직접 참조하는 테이블과 컬럼은 하나의 `schema.sql`에서 확인할 수 있습니다. 상세 매핑은 [데이터베이스 스키마와 DAO 매핑](database-schema.md)에 정리했습니다.

## 3. 신체정보 건너뛰기 화면 전환

### 문제

회원가입 후 신체정보 입력을 건너뛰면 사용자 메인 화면이 아니라 로그인 화면으로 돌아갔습니다.

### 원인

`BodyInfoSetPanel`의 Skip 동작이 CardLayout에서 로그인 패널 이름을 호출하고 있었습니다.

### 해결

Skip 동작이 사용자 메인 패널 이름을 호출하도록 화면 전환 대상을 수정했습니다.

### 결과

신체정보 입력 여부와 관계없이 회원가입 이후 사용자 흐름을 이어갈 수 있게 되었습니다.

## 4. 식단 헤더와 상세 음식의 부분 저장 가능성

### 문제

식단 헤더와 여러 음식 상세가 서로 다른 연결과 저장 단계에서 처리되면 중간 실패 시 일부 데이터만 남을 수 있었습니다.

### 원인

`meal`과 `meal_log` 저장이 하나의 트랜잭션 경계로 묶여 있지 않았습니다.

### 해결

- 같은 JDBC `Connection`으로 식단 헤더와 상세 음식을 저장했습니다.
- 모든 상세 저장이 성공한 경우에만 `commit`했습니다.
- 실패 시 `rollback`하고 성공 안내와 화면 이동을 차단했습니다.

### 결과

식단 한 건의 저장 단위를 명확하게 하고 부분 저장 위험을 줄였습니다.

## 5. DB 접속정보와 공개 저장소 분리

### 문제

로컬 DB 접속 설정은 실행에 필요하지만 비밀번호를 포함할 수 있어 공개 저장소에 올릴 수 없습니다.

### 해결

- 실제 설정인 `src/db.properties`를 `.gitignore`에 등록했습니다.
- 키 구조만 제공하는 `src/db.properties.example`을 유지했습니다.
- 공개 SQL에는 실제 관리자 계정과 비밀번호를 포함하지 않았습니다.

### 결과

실행에 필요한 설정 방법을 제공하면서 환경별 접속정보는 로컬에서만 관리합니다.
