## 2026-07-13 / 회원정보 저장 원자성 보완

- 원인: 수정 전 패널은 현재 비밀번호 검증을 업데이트 DAO 내부에 위임한 채 먼저 임시 `UserBean`을 만들고 업데이트 가능한 메서드를 호출했다. DAO도 새 비밀번호 배열 길이에 따라 일반 정보 저장으로 우회할 수 있었으며, 실패 시 수정 폼의 일반 입력값을 세션 값으로 복구하지 않았다. 저장 실패 경계를 패널과 DAO에서 명시적으로 분리하지 않은 구조였다.
- 기존 DML 확인: 비밀번호 변경 분기 자체는 일반 정보와 비밀번호를 하나의 조건부 `UPDATE`로 처리하고 있어 두 `UPDATE` 사이의 부분 저장 경로는 없었다. DB에 접속하지 않았으므로 사용자 실행에서 확인된 DB 상태는 정적으로 재현하지 않았다.
- 검증 우선: 이름·이메일·전화번호는 지역변수에만 보관한다. 비밀번호 변경 요청이면 현재 비밀번호 존재 확인, 읽기 전용 현재 비밀번호 검증, 새 비밀번호 일치, 기존 비밀번호 규칙 검증 순으로 처리한다. 모든 검증이 끝난 뒤에만 업데이트용 `UserBean`을 만든다.
- 명시적 저장 경로: 일반 정보만 저장하는 `updateBasicUserInfo`와 일반 정보·비밀번호를 함께 저장하는 `updateUserInfoAndPassword`를 분리했다. 비밀번호 포함 저장에는 일반 정보만 저장하는 대체 경로가 없다.
- 원자성: `updateUserInfoAndPassword`가 하나의 JDBC `Connection`에서 현재 저장값을 다시 검증하고 이름·전화번호·이메일·새 비밀번호 해시를 단일 조건부 `UPDATE`로 저장한다. 단일 DML이므로 두 업데이트 사이의 commit/rollback 구간이 없으며, 저장 비밀번호가 검증 이후 바뀌면 조건절에 의해 전체 실패한다.
- 세션 반영: 로그인 시 `LoginManager`와 `UserSessionManager`가 같은 일반 사용자 `UserBean` 참조를 보관한다. DB 저장 결과가 `SUCCESS`인 경우에만 이 객체의 이름·이메일·전화번호 setter를 호출한다.
- 실패 복구: 현재 비밀번호 누락·불일치 또는 DAO 오류이면 일반 입력 필드를 기존 세션 값으로 되돌리고 비밀번호 3개 필드와 보기 상태를 초기화한다. `MyPagePanel`은 변경되지 않은 세션 객체를 다시 읽는다.

### CRUD 이후 클릭 응답 수정

- 원인: 공지 카드와 버튼 리스너는 목록을 다시 만들 때마다 새 컴포넌트에 한 번씩 정상 등록되고 있었다. 잔존 glass pane·overlay·중복 패널·비활성 패널도 없었다. 실제 문제는 `ActionListener`가 실행되는 Swing EDT에서 공지 등록·수정·삭제와 복귀 직후 전체 목록·상세·첨부 조회까지 동기로 수행해, DB 작업이 끝날 때까지 이벤트 큐가 다음 마우스 입력을 처리하지 못한 구조였다.
- 목록 갱신: 사용자·관리자 목록 조회를 `SwingWorker.doInBackground()`로 이동하고 `done()`에서만 기존 카드 제거, 새 카드·리스너 구성, `revalidate()`, `repaint()`, 스크롤 상단 이동을 수행한다. 연속 갱신은 세대 번호로 이전 결과가 최신 화면을 덮지 않게 했다.
- 화면 전환: `MainAdminPanel.showPanel()`은 EDT에서 실제 등록 패널 인스턴스의 visible·enabled 상태를 먼저 확정한 후 목록 갱신을 시작하며, 마지막에 컨테이너를 다시 검증하고 그린다.
- CRUD: 등록·수정·단일 삭제·다중 삭제를 백그라운드에서 수행하고 완료 콜백에서만 안내창과 화면 전환을 처리한다. 처리 중 중복 실행을 막기 위해 관련 버튼만 잠시 비활성화하고 모든 성공·실패·예외 경로의 `finally`에서 복구한다.
- 상세 진입: 사용자·관리자 상세와 관리자 수정 데이터 조회도 백그라운드로 분리했다. 새 요청이 들어오면 이전 조회 결과를 무시하며, 새 데이터 적용 전 기존 상세·수정 상태를 초기화한다.
- 첨부 다운로드: 파일 조회·복사도 백그라운드로 이동하고 해당 파일 버튼만 작업 중 비활성화한 뒤 `finally`에서 다시 활성화한다.
- 이벤트 유지: 목록을 새로 구성할 때마다 `createNoticeCard()`가 새 리스너를 정확히 한 번 등록한다. 관리자 체크박스에는 상세 리스너를 등록하지 않는 기존 분리를 유지했다.

### 첨부파일 저장 실패 수정

- 원인: 정적 스키마와 기존 비교 기록의 `notice_files` 계약은 `notice_id`, `file_name`, `file_path`, `uploaded_time`을 사용한다. 기존 `NoticeFileDAO`는 존재하지 않는 `admin_id`, `file_data`, `upload_time` 컬럼에 BLOB을 INSERT하고 다운로드에서도 `file_data`를 조회해 첨부 저장·열기가 실패했다.
- 계약 정렬: DB 구조를 변경하지 않고 `NoticeFileDAO`를 기존 path 방식 컬럼에 맞췄다. `NoticeDAO`의 공지 조회·등록·수정·삭제 SQL은 변경하지 않았다.
- 파일 저장: `JFileChooser`의 `File`을 `Path`로 변환하고 사용자 홈 기준 앱 전용 첨부 디렉터리를 `Files.createDirectories()`로 준비한 뒤 `Files.copy()`로 복사한다. 외부 절대경로 하드코딩은 사용하지 않는다.
- 파일명 처리: `file_name`에는 표시용 원본 파일명을 기존 255자 계약 안에서 보존하고, `file_path`에는 충돌과 긴 경로를 피하는 UUID 기반 저장명만 기록한다. 한글·공백·괄호는 표시 이름에 유지되며 저장명과 분리된다.
- Windows 대응: 다운로드 파일명에서 운영체제 금지 문자·제어문자·예약 이름을 안전하게 치환한다. 같은 이름이 있으면 번호 접미사를 붙이고 다운로드 디렉터리가 없으면 생성한다.

## 2026-07-14 / UI 테마 렌더링·여백 보정

### 사용자 확인 결과와 보정 범위

- 일일 현황, 식단 홈, 음식 검색·상세, 목표 달성, 캘린더, 운동 카테고리, 내 정보, 회원정보 수정, 사용자 공지 목록·상세는 채택 후보 화면으로 유지하고 구조를 변경하지 않았다.
- 보정 범위는 운동 목록과 신체정보 수정의 명백한 렌더링 오류, 담은 음식 빈 상태·목표 수정·운동 기록 입력·신체정보 수정의 과도한 여백으로 제한했다.
- `AppTheme`, `RoundedComponent`, 공통 헤더·하단 내비게이션, `MainFrame` 크기와 화면 전환 이름은 변경하지 않았다.

### 렌더링 오류 원인과 수정

- 운동 목록 원인: 일반 `JPanel` 목록이 `JScrollPane` viewport 폭을 추적하지 않았고, 축소 가능한 wrapper 안에서 360px 고정 카드를 `FlowLayout.CENTER`로 배치했다. wrapper가 카드보다 좁아지면 카드가 음수 x 방향으로 밀려 카드 일부와 왼쪽 텍스트가 잘리고, 짧은 텍스트는 흰 카드처럼 보일 수 있는 구조였다.
- 운동 목록 수정: 목록 패널이 `Scrollable`을 구현해 viewport 폭을 추적하게 하고 가로 스크롤을 금지했다. 고정 wrapper를 제거하고 모든 카드를 viewport 가용 폭에 맞춰 늘어나게 했으며 운동명은 최대 두 줄로 표시한다. 카테고리 목록과 검색 결과는 기존과 동일하게 하나의 카드 생성 메서드를 사용한다.
- 신체정보 원인: `RoundedComponent` 카드의 내부 `BorderLayout`에 제목·설명·입력 필드·버튼을 제약 없이 추가했다. 마지막 `CENTER` 컴포넌트인 저장 버튼이 카드 전체로 확대되어 버튼의 진한 초록 배경이 제목 영역에 그려지고 실제 버튼 내용은 의도한 bounds 밖으로 밀렸다.
- 신체정보 수정: 해당 패널 내부 카드만 null layout의 일반 `JPanel`로 교체했다. 제목은 투명한 일반 제목으로 유지하고 `내 정보로` 버튼과 같은 헤더 행에 배치했으며, 기존 `ActionListener`, 입력값 로드·검증, `BodyInfoDAO.saveBodyInfo()`와 성공·실패 안내를 유지했다.

### 화면별 여백 보정

- 담은 음식 빈 상태: 기존 150px 빈 상태 카드는 유지하고 빈 목록에서는 스크롤 영역을 콘텐츠 높이에 맞춘 뒤 버튼을 16px 아래에 배치한다. 음식이 생기면 기존 524px 스크롤 영역과 하단 버튼 위치를 복구하며 검색 탭 복귀 시에도 기본 bounds를 다시 적용한다.
- 목표 수정: 저장 버튼 아래 24px만 남도록 폼 카드 높이를 500px에서 460px로 줄였다.
- 운동 기록 입력: 저장 버튼 아래 24px만 남도록 카드 높이를 570px에서 458px로 줄였다.
- 신체정보 수정: 제목·설명 간격 6px, 설명과 첫 입력 간격 18px, 입력 행 간격 16px, 마지막 입력과 저장 버튼 간격 26px, 저장 버튼 아래 24px로 맞추고 카드 높이를 448px로 줄였다.
