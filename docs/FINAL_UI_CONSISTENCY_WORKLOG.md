## 2026-07-13 / 회원정보 저장 원자성 보완

- 원인: 수정 전 패널은 현재 비밀번호 검증을 업데이트 DAO 내부에 위임한 채 먼저 임시 `UserBean`을 만들고 업데이트 가능한 메서드를 호출했다. DAO도 새 비밀번호 배열 길이에 따라 일반 정보 저장으로 우회할 수 있었으며, 실패 시 수정 폼의 일반 입력값을 세션 값으로 복구하지 않았다. 저장 실패 경계를 패널과 DAO에서 명시적으로 분리하지 않은 구조였다.
- 기존 DML 확인: 비밀번호 변경 분기 자체는 일반 정보와 비밀번호를 하나의 조건부 `UPDATE`로 처리하고 있어 두 `UPDATE` 사이의 부분 저장 경로는 없었다. DB에 접속하지 않았으므로 사용자 실행에서 확인된 DB 상태는 정적으로 재현하지 않았다.
- 검증 우선: 이름·이메일·전화번호는 지역변수에만 보관한다. 비밀번호 변경 요청이면 현재 비밀번호 존재 확인, 읽기 전용 현재 비밀번호 검증, 새 비밀번호 일치, 기존 비밀번호 규칙 검증 순으로 처리한다. 모든 검증이 끝난 뒤에만 업데이트용 `UserBean`을 만든다.
- 명시적 저장 경로: 일반 정보만 저장하는 `updateBasicUserInfo`와 일반 정보·비밀번호를 함께 저장하는 `updateUserInfoAndPassword`를 분리했다. 비밀번호 포함 저장에는 일반 정보만 저장하는 대체 경로가 없다.
- 원자성: `updateUserInfoAndPassword`가 하나의 JDBC `Connection`에서 현재 저장값을 다시 검증하고 이름·전화번호·이메일·새 비밀번호 해시를 단일 조건부 `UPDATE`로 저장한다. 단일 DML이므로 두 업데이트 사이의 commit/rollback 구간이 없으며, 저장 비밀번호가 검증 이후 바뀌면 조건절에 의해 전체 실패한다.
- 세션 반영: 로그인 시 `LoginManager`와 `UserSessionManager`가 같은 일반 사용자 `UserBean` 참조를 보관한다. DB 저장 결과가 `SUCCESS`인 경우에만 이 객체의 이름·이메일·전화번호 setter를 호출한다.
- 실패 복구: 현재 비밀번호 누락·불일치 또는 DAO 오류이면 일반 입력 필드를 기존 세션 값으로 되돌리고 비밀번호 3개 필드와 보기 상태를 초기화한다. `MyPagePanel`은 변경되지 않은 세션 객체를 다시 읽는다.
- 비밀번호 안전: 세 비밀번호는 `char[]`로 전달하고 `finally`에서 모두 `Arrays.fill()`로 지운다. PBKDF2 구현·반복 횟수·검증 정책은 변경하지 않았다.
